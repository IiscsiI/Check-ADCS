Fonctionnement et périmètre de sécurité

Cette interface est un visualiseur hors ligne : elle ne se connecte jamais à l'autorité de certification. Elle lit un export JSON produit par le collecteur Collecte-ADCS.ps1, qui interroge la base de la CA en lecture seule (interface COM ICertView, RPC), avec un compte disposant uniquement du droit Read (ou du rôle Auditor si la séparation des rôles est activée).

Collecte planifiée recommandée

powershell.exe -ExecutionPolicy AllSigned -File Collecte-ADCS.ps1 -Config "SERVEUR\Nom CA" -Html visualiseur-adcs.html -Sortie \\partage\pki

  • -Depuis (Get-Date).AddYears(-3) pour limiter la profondeur sur les bases volumineuses ;
  • -Html produit un fichier auto-porteur unique (données embarquées) à déposer sur un partage DSI ;
  • -CheminCRL pour pointer vers un CDP HTTP si le partage CertEnroll n'est pas accessible.

Actions (révocation, approbation, refus)

L'outil n'exécute aucune action. Il pré-génère des commandes certutil à copier, à exécuter par un administrateur de la CA habilité, depuis un poste d'administration, afin que chaque action reste un geste conscient, tracé dans les journaux de la CA et imputable.

Après une révocation, pensez à publier une nouvelle CRL (certutil -config "…" -crl) pour que la révocation soit effective côté clients avant la prochaine publication planifiée.

Révocation de masse

Cochez les certificats concernés (ou la case d'en-tête pour toute la vue filtrée), choisissez le motif dans la barre d'actions, puis « Préparer la révocation ». Après relecture du récapitulatif, préférez le script PowerShell généré : il exige la saisie du nombre exact de certificats pour confirmer, journalise chaque révocation dans un transcript, dresse un bilan succès/échecs, et propose la publication d'une nouvelle CRL. Les certificats expirés et déjà révoqués ne sont pas sélectionnables (les CRL ne portent pas les certificats expirés). Le script généré est à relire puis signer avant exécution, conformément à votre politique AllSigned.

Doublons à nettoyer

Un certificat est marqué doublon lorsqu'un certificat plus récent, valide, existe pour le même nom commun et le même modèle — typiquement après un ré-enrôlement. Les certificats à courte durée de vie (validité ≤ 60 jours, comme la signature de réponses OCSP) sont exclus de cette détection : leur chevauchement est normal et leur révocation serait dangereuse. Avant toute révocation d'un doublon, vérifiez que le certificat le plus récent est bien déployé et utilisé (poste ayant récupéré son nouveau certificat, service rechargé) : le motif 6 (suspension, réversible via certutil -revoke <série> unrevoke) est une option prudente pour un premier lot.

Partager une vue

Les filtres (recherche, état, modèle) s'encodent dans l'URL : copiez l'adresse pour envoyer exactement la même vue à un collègue ouvrant le même fichier. Raccourci : / place le curseur dans la recherche.

États affichés

  • Actif certificat émis, valide, expirant dans plus de 90 jours ;
  • Expire ≤ N j certificat valide expirant dans 90 jours ou moins ;
  • Expiré date de fin de validité dépassée ;
  • Révoqué présent dans la base avec disposition 21.

Cet outil n'exécute aucune action sur la CA. Vérifiez chaque ligne ci-dessous : une révocation (hors motif 6) est définitive. Le script généré exige une confirmation explicite, journalise dans un transcript, et propose la publication d'une nouvelle CRL en fin d'exécution.

Recommandé pour un lot : le script PowerShell (confirmation par saisie du nombre exact, transcript de journalisation, bilan par certificat). Les commandes brutes conviennent pour 2–3 certificats.
